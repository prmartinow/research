## A. CPU-only recommendation

For CPU-only inference of reranking models, replace larger models like **Qwen3-Embedding-8B** with a small code-focused embedder and replace **Qwen3-Reranker-8B** with a tiny bounded cross-encoder. Keep lexical/BM25/Zoekt, symbol, SCIP/tree-sitter, and graph expansion as the main recall system.

Recommended first pass:

1. **Embedding sidecar:** evaluate **CodeRankEmbed 137M**, **SFR-Embedding-Code-400M_R**, **C2LLM-0.5B**, and **Jina embeddings v2 base code** first. Add **CodeSage-small-v2** as an Apache-licensed legacy baseline and **SweRankEmbed-Small** specifically for issue-to-file/function localization.
2. **Bounded reranking:** start with **Ettin-reranker-17M**, then **Ettin-reranker-32M/68M** if latency allows. Use **Jina-reranker-v1-tiny/turbo** and **MiniLM-L6-v2** as generic baselines. Use **Qwen3-Reranker-0.6B** only as a slow top-20 quality experiment, not as the default CPU path.
3. **Avoid for full CPU corpus indexing:** Qwen3-Embedding-8B, Qwen3-Reranker-8B, BGE-Code-v1 if indexing throughput matters, CodeRankLLM/SweRankLLM/CoREB-Reranker, and whole-corpus ColBERT-style multi-vector indexing unless you deliberately accept a separate storage/update-cost profile.

The evidence is mixed and task-dependent. CoIR favors several code-specific embedders over older CodeBERT/UniXcoder-style baselines, while CoREB’s 2026 reranking study found that off-the-shelf rerankers can **hurt** code-search ranking unless trained for code reranking. ([My HomePage][1])

## E. Leaderboard and benchmark evidence summary

### Embeddings

**CoIR** is the most useful public embedding benchmark for code information retrieval. It covers multiple code IR tasks and shows newer code-focused embedders ahead of older UniXcoder/BGE-M3-style baselines. In the visible CoIR table, **SFR-Embedding-Code-2B_R** ranks first, **SFR-Embedding-Code-400M_R** ranks highly among smaller models, and **CodeRankEmbed 137M** is competitive for its size. ([My HomePage][1])

**CoREB** is more directly relevant to code-search pipelines because it evaluates first-stage embedding and reranking. Its May 2026 evidence is important: among ~0.5B-class models in the reported table, **C2LLM-0.5B** and **Jina-code-emb-0.5B** outperform Qwen3-0.6B and Qwen3-8B on CoREB’s overall nDCG/recall. This does not prove they win on your repositories, but it is strong enough to include them in the CPU shortlist. ([arXiv][14])

**CORE-Bench** is the best-aligned public benchmark for agentic repository retrieval. It explicitly targets code understanding, issue-to-edit localization, and broader-context retrieval across repos, and it reports that current embedding retrievers still fall short while in-domain supervised fine-tuning helps. Use it to avoid overfitting to function-level CodeSearchNet-style retrieval. ([arXiv][27])

**RepoBench, CrossCodeEval, RepoEval/RepoCoder** are useful for cross-file completion and context retrieval, but they are not clean CPU-embedding leaderboards. Use them to test whether retrieved context actually helps downstream code intelligence, not just whether a vector model has high text-to-code nDCG. CrossCodeEval also reinforces that BM25 remains a strong code baseline in cross-file settings. ([arXiv][28])

### Reranking

The correct reranking benchmark class is **code-search candidate reranking**: text-to-code, code-to-text, code-to-code, issue-to-file/function, and repository-level candidate ranking. Generic MS MARCO/BEIR passage reranking is only a latency/architecture baseline.

**CoREB is the closest public reranking benchmark**, and its conclusion is cautious: no off-the-shelf reranker was net-positive across all three code-search tasks in their reported setup; the positive reranker was **CoREB-Reranker**, a Qwen3-4B LoRA trained on 3.1M code-search examples, which is not CPU-suitable for your local path. ([arXiv][14])

For CPU-only production, this means the reranker shortlist should be chosen by **latency and in-domain delta**, not by generic leaderboard score. Ettin is attractive because it gives a modern, very small cross-encoder family with explicit CPU throughput measurements. It is not code-specific; it must be tested on your code-search labels. ([Hugging Face][16])

## F. Final CPU shortlist to evaluate first

### Embeddings, ranked by expected quality-per-indexing-hour

1. **CodeRankEmbed 137M** — best first test: small, MIT, code-specific, strong reported CoIR/CodeSearchNet for its size.
2. **SFR-Embedding-Code-400M_R** — likely strong quality; evaluate if research-use license and 400M CPU throughput are acceptable.
3. **C2LLM-0.5B** — best newer 0.5B-class quality candidate from CoREB evidence; heavier, but worth testing.
4. **Jina embeddings v2 base code** — fast Apache-licensed code baseline despite being older/deprecated.
5. **CodeSage-small-v2 130M** — clean Apache legacy baseline.
6. **Jina code embeddings 0.5B** — evaluate if CC-BY-NC and 0.5B CPU cost are acceptable.
7. **LateOn-Code-edge / LateOn-Code** — separate late-interaction experiment, not a direct replacement for single-vector indexing.
8. **SweRankEmbed-Small** — include for issue-localization workflows only.
9. **potion-code-16M** — speed-floor baseline to compare against BM25/hybrid.
10. **Qwen3-Embedding-0.6B** — Qwen sanity check only; do not prioritize over the code-specific models above.
11. **BGE-Code-v1** — quality reference, but likely too large for many-repo CPU exhaustive indexing.

### Rerankers, ranked for bounded top-20/50 CPU use

1. **Ettin-reranker-17M-v1** — default CPU reranker baseline.
2. **Ettin-reranker-32M-v1** — likely better quality if latency is still acceptable.
3. **Ettin-reranker-68M-v1** — upper CPU mode for top-20 or slower top-50.
4. **Jina-reranker-v1-tiny/turbo** — small Apache baseline with long input support.
5. **MiniLM-L6-v2 cross-encoder** — older generic sanity baseline.
6. **Custom CodeBERT-base cross-encoder** — train on your own PR/issue/query labels; this is the most likely path to a genuinely code-aware CPU reranker.
7. **Qwen3-Reranker-0.6B** — slow top-20/offline comparison only; keep out of default interactive CPU path unless your eval proves a large gain.

### Minimum evaluation matrix

Run the model sweep against your own chunking and **maxp passage→file/symbol aggregation**:

* Retrieval configs: lexical only, symbol/graph only, dense only, lexical+dense RRF, lexical+dense+symbol, +graph expansion.
* Reranker configs: no reranker, Ettin-17M, Ettin-32M, Jina-v1-tiny, MiniLM, Qwen3-0.6B on top 20/50.
* Metrics: chunk Recall@50/100, file/symbol Recall@20/50, nDCG@10, MRR, target edited-file recall, p95 query latency, chunks/sec indexing, embeddings GB per million chunks, and incremental re-embedding time per commit.

The first two models I would actually run are **CodeRankEmbed for embeddings** and **Ettin-reranker-17M for bounded reranking**. Add **SFR-400M_R** and **C2LLM-0.5B** immediately after to establish the quality/cost frontier.
